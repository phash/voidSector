01. Monitor Programs

The multi-monitor system provides 12 distinct programs, each rendered inside a bezel frame. Monitors can be assigned to sidebar slots (left/right, 2 per side), displayed in the main area (split or fullscreen mode), or accessed via mobile tabs. Monitor IDs are defined in packages/shared/src/constants.ts as the MONITORS constant.

ID COMPONENT DESCRIPTION SIDEBAR MAIN
NAV-COM NavComScreen (inline in GameScreen) Primary navigation. Contains RadarCanvas, SectorInfo, StatusBar, NavControls. In split mode, wrapped in MonitorBezel with zoom/pan knobs, BookmarkBar, and DetailViewOverlay. Features pulsing player border, staleness rendering, jump animation, feature dots (jumpgate/scan events), and bookmark markers. Left only Default (split)
LOG EventLog Scrollable event log with timestamped entries. Shows navigation, scan results, combat outcomes, and system messages. Available only on the left sidebar. Left only —
SHIP-SYS ShipSysScreen (inline in GameScreen) Three sub-tabs: SCHEMATIC (ASCII ship diagram per hull type with slot labels, module breakdown, ship stats grid, color profile selector), MODULE (ModulePanel for equipping/unequipping modules), HANGAR (HangarPanel for ship management). Hull types: scout, freighter, cruiser, explorer, battleship. Both —
MINING MiningScreen Mining operations. Shows rig status, current yield, resource extraction progress. Features auto-mining with AP consumption and cargo capacity tracking. Both Yes
CARGO CargoScreen Cargo inventory display. Shows ore, gas, crystal, data slates, artefacts. Capacity indicator with color-coded warnings (green/yellow/red). Includes transfer and jettison controls. Both Yes

Monitor Routing

Monitors are assigned to layout slots via Zustand state. The renderScreen(monitorId) function in GameScreen.tsx maps each MONITORS.* constant to its React component. Three routing arrays control availability: 

LEFT_SIDEBAR_MONITORS:  LOG, SHIP-SYS, MINING, CARGO, COMMS, BASE-LINK, TRADE, FACTION, QUESTS, TECH, QUAD-MAP
RIGHT_SIDEBAR_MONITORS: SHIP-SYS, MINING, CARGO, COMMS, BASE-LINK, TRADE, FACTION, QUESTS, TECH, QUAD-MAP
MAIN_MONITORS:          NAV-COM, MINING, CARGO, COMMS, BASE-LINK, TRADE, FACTION, QUESTS, TECH, QUAD-MAP

Note: LOG is left-sidebar only. NAV-COM is main-area only (split or fullscreen). SHIP-SYS is sidebar-only (not available in main area).

Mobile Tabs

On screens below 1024px, the desktop layout is hidden and a bottom tab bar is shown. 8 tabs are exposed (not all 12 monitors): 

NAV (NAV-COM)  |  SHIP (SHIP-SYS)  |  MINE (MINING)  |  CARGO (CARGO)
COMMS (COMMS)  |  BASE (BASE-LINK)  |  TECH (TECH)    |  QUAD (QUAD-MAP)

LOG, TRADE, FACTION, and QUESTS are not accessible via mobile tabs. Each tab shows an icon and a short label. Active tab has a bottom highlight bar with glow effect. Alert state triggers a blink animation.

02. Bezel Components

The bezel system simulates physical CRT monitor hardware. Two bezel variants exist: a full-featured MonitorBezel for the main area and a simplified SidebarBezel for sidebar slots. This creates a visual divergence documented in the UX Report.

MonitorBezel
packages/client/src/components/MonitorBezel.tsx
Props: children: ReactNode, monitorId: string
Full CRT bezel frame wrapping the main nav monitor. Renders all CRT effects (scanlines, flicker, glitch bars, vignette, noise). Features:
• Left bezel side: Vertical monitor ID label, PAN knob (range -20..20, step 1), BookmarkBar
• Right bezel side: Status LEDs (via useMonitorLeds), ON/OFF toggle switch, ZOOM knob (range 0..3, step 1), POWER button
• Bottom bezel: BRIGHTNESS knob (range 0.5..1.5, step 0.1), AUTO/MANUAL follow button (NAV-COM only), [?] legend button
• Screen area: CRT wrapper with scanlines, flicker, glitch bars, vignette, optional noise (nebula). Content wrapped in CrtErrorBoundary.
• Program-switch flicker: 300ms crt-switch-flicker animation when children identity changes.
• Power toggle: Turns monitor on/off. Off state shows "DISPLAY OFF" overlay.
• Noise: Activated when monitorId === 'NAV-COM' and currentSector.type === 'nebula'.
SidebarBezel
packages/client/src/components/SidebarBezel.tsx
Props: children: ReactNode, monitorId: string, alert?: boolean
Simplified bezel for sidebar slots. Notable differences from MonitorBezel:
• Only renders scanlines and vignette (no flicker, no glitch bars, no noise)
• Left bezel side only: vertical label (0.45rem) + LED dots
• No right bezel side, no bottom controls
• No zoom/pan/brightness knobs
• No BookmarkBar, no power toggle
• Alert prop triggers bezel-alert-pulse animation on border
• Auto-clears alert when monitor becomes visible (useEffect on mount)
• Uses inline styles for bezel side (minWidth: 20, padding: '4px 2px')
ChannelButtons
packages/client/src/components/ChannelButtons.tsx
Props: slotIndex: 0 | 1, side: 'left' | 'right', monitors: MonitorId[]
Vertical strip of channel selector buttons rendered adjacent to sidebar bezels. Each button represents a monitor that can be loaded into the parent sidebar slot.
• Active state: inverted colors (dark text on primary background)
• Alert state: channel-blink animation (1s ease-in-out infinite) when monitor has pending alert
• Label abbreviation map: LOG→LOG, SHIP-SYS→SYS, MINING→MIN, CARGO→CRG, COMMS→COM, BASE-LINK→BAS
• Fallback: first 3 chars of monitor ID
• Button size: 64x48px, font 0.7rem, letter-spacing 0.05em
BezelKnob
packages/client/src/components/BezelKnob.tsx
Props: label: string, value: number, min: number, max: number, step?: number, onChange: (value: number) => void
Vertical range slider styled as a CRT hardware knob. Used for BRIGHTNESS, ZOOM, and PAN controls.
• Rendered as a vertical <input type="range"> (80px tall, 20px wide)
• writingMode: 'vertical-lr', direction: 'rtl' for vertical orientation
• accentColor: var(--color-primary) for slider track/thumb
• Displays label above (0.55rem, dim) and current value below (0.55rem, primary)
• Value display precision: 1 decimal if step < 1, else integer
• All styling is inline (no CSS class)
LedDot
packages/client/src/components/MonitorLeds.tsx
Props: led: LedConfig where LedConfig = { label: string, color: 'green' | 'yellow' | 'red' | 'gray', blink?: boolean }
Single status LED indicator with label. Used in both MonitorBezel and SidebarBezel.
• Colors: green=#00FF88, yellow=#FFB000, red=#FF3333, gray=#444
• Active LEDs get box-shadow: 0 0 4px glow
• Blink uses bezel-alert-pulse 1.5s infinite animation
• LED diameter: 6px, label font: 0.6rem at 0.6 opacity
• All styling is inline (no CSS class)
useMonitorLeds (Hook)
packages/client/src/components/MonitorLeds.tsx
Signature: useMonitorLeds(monitorId: string): LedConfig[]
Returns an array of LED configurations based on the current monitor and game state.
• NAV-COM: SYS (green), NAV (yellow if autopilot active, else green)
• SHIP-SYS: PWR (green), FUEL (red if <20%, yellow if <50%, else green)
• MINING: RIG (green if mining active, gray if idle)
• CARGO: CAP (red if full, yellow if >70%, else green)
• COMMS: SIG (green), MSG (yellow+blink if alert), SOS (red+blink if distress calls)
• QUESTS: QST (yellow+blink if alert, else gray)
• Default: SYS (green) for all other monitors
BookmarkBar
packages/client/src/components/BookmarkBar.tsx
Props: none (reads from store)
Vertical bookmark navigation strip rendered inside the MonitorBezel left side.
• Two fixed buttons: HOME (jumps to 0,0) and SHIP (centers on player position)
• 5 custom bookmark slots (1-5), showing label or coordinates
• Left-click: pans view to bookmarked sector and selects it
• Right-click: clears the bookmark (sends clearBookmark to server)
• Empty slots shown at 0.3 opacity, disabled
• Style: vs-btn class, 0.55rem font, 2px 4px padding
CrtErrorBoundary
packages/client/src/components/CrtErrorBoundary.tsx
Props: children: ReactNode, monitorId?: string
React Error Boundary that wraps monitor content inside MonitorBezel. Catches render errors and displays a CRT-themed "SYSTEM FAULT" screen.
• Phase 1 (2 seconds): "SYSTEM FAULT" in red with flicker animation, glitch bar, "REBOOTING..." message
• Phase 2: "MONITOR OFFLINE" with [REBOOT] button
• Logs error to console as [CRT:{monitorId}]
• Reboot resets error state, re-renders children

03. CRT Effects

CRT effects are layered via absolute-positioned overlays inside the .crt-wrapper container. Each effect has a specific z-index for correct stacking. All effects are defined in packages/client/src/styles/crt.css. Effects respect prefers-reduced-motion: reduce by disabling animations.

Scanlines

SCANLINES
.crt-scanlines
Z-index: 10
Implementation: Dual-layer background gradient:
1. Horizontal scanlines: alternating transparent/dark rows at 3px pitch
2. Vertical RGB sub-pixel simulation: faint red/green/blue bands at 3px pitch
CSS: 
background:
  linear-gradient(
    rgba(18, 16, 16, 0) 50%,
    rgba(0, 0, 0, 0.25) 50%
  ),
  linear-gradient(
    90deg,
    rgba(255, 0, 0, 0.03),
    rgba(0, 255, 0, 0.02),
    rgba(0, 0, 255, 0.03)
  );
background-size: 100% 3px, 3px 100%;

Flicker

FLICKER
.crt-flicker
Z-index: 11
Implementation: Semi-transparent dark overlay with rapid opacity oscillation
Animation: crt-flicker 0.15s infinite — cycles opacity between 0.18 and 0.43
Note: Only present in MonitorBezel, absent from SidebarBezel
Reduced motion: animation: none

Glitch Bars

GLITCH BAR
.crt-glitch-bar / .crt-glitch-bar-2
Z-index: 13
Implementation: Horizontal distortion stripes that scroll top-to-bottom
Bar 1: 3px tall, amber-tinted gradient, 12s cycle, hue-rotate(20deg) effect
Bar 2: 2px tall, teal-tinted gradient, 18s cycle with -7s delay, hue-rotate(-15deg)
Note: Only present in MonitorBezel, absent from SidebarBezel
Reduced motion: animation: none

Vignette

VIGNETTE
.crt-vignette
Z-index: 9
Implementation: Radial gradient from transparent center to dark edges
CSS: radial-gradient(ellipse at center, transparent 75%, rgba(0, 0, 0, 0.06) 100%)
Note: Present in both MonitorBezel and SidebarBezel

Noise

NOISE (NEBULA)
.crt-noise
Z-index: 14
Implementation: SVG fractal noise pattern via inline data URI, animated with stepped position shifts
Animation: crt-noise 0.15s steps(1) infinite — shifts background-position in 10 keyframes
Opacity: 0.08 base
Trigger: Only shown when monitorId === 'NAV-COM' and player is in a nebula sector
Purpose: Simulates signal interference in nebula environments

Program-Switch Flicker

.crt-switch-flicker
Animation: crt-switch-flicker 0.3s ease-out forwards
Keyframes: Rapid brightness/opacity oscillation simulating a CRT tube switching input
0%: normal → 10%: blackout+bright → 20%: normal+dim → 35%: partial+bright → 50%: normal → 100%: normal
Trigger: Applied to .crt-content when MonitorBezel detects children identity change

Sidebar CRT Collapse/Expand

.sidebar-crt-collapse / .sidebar-crt-expand
Animation: 250ms ease-in/ease-out
Effect: scaleX from 1 to 0 (collapse) or 0 to 1 (expand), with brightness flash at midpoint
Transform origin: left center for left sidebar, right center for right sidebar
Purpose: Simulates CRT tube power-on/power-off when toggling sidebar visibility

04. Color Profiles

Four color profiles are defined in packages/client/src/styles/themes.ts. The active profile is applied by setting CSS custom properties --color-primary and --color-dim on document.documentElement (in GameScreen via useEffect). Profile selection is persisted to localStorage key vs-color-profile. Default: Amber Classic.

AMBER CLASSIC
primary: #FFB000
dim: rgba(255, 176, 0, 0.6)
GREEN PHOSPHOR
primary: #00FF66
dim: rgba(0, 255, 102, 0.6)
ICE BLUE
primary: #00CCFF
dim: rgba(0, 204, 255, 0.6)
HIGH CONTRAST
primary: #FFFFFF
dim: rgba(255, 255, 255, 0.6)

CSS Custom Properties (Default / Amber Classic)

:root {
  --color-primary: #FFB000;          /* Main text, borders, active elements */
  --color-dim: rgba(255, 176, 0, 0.6); /* Secondary text, labels */
  --color-bg: #050505;               /* Background */
  --color-danger: #FF3333;           /* Error, combat, pirate encounters */
  --color-bezel: #1a1a1a;            /* Bezel panel background */
  --color-bezel-light: #2a2a2a;      /* Bezel dividers */
  --color-led-active: #FFB000;       /* Active LED glow */
  --color-led-off: #3a3000;          /* Inactive LED */
  --font-mono: 'Share Tech Mono', 'Courier New', Courier, monospace;
}

Note: Only --color-primary and --color-dim are updated by the color profile system. Hardcoded colors like --color-danger (#FF3333), --color-led-active, and bezel background colors remain fixed regardless of profile. LED colors are also hardcoded: green=#00FF88, yellow=#FFB000, red=#FF3333.

05. Canvas Renderers

Two canvas renderers power the game’s map displays. Both use the useCanvas hook which manages DPR-aware sizing via ResizeObserver and drives rendering through requestAnimationFrame loops.

RadarRenderer
packages/client/src/canvas/RadarRenderer.ts
Sector-level radar grid. The main visual display of the game world.

Zoom levels (0-4):
ZoomCell SizeFontFeatures
048x38px12pxOverview, no coords per cell
164x50px14pxCoordinates, sector labels
280x64px16pxFeature dots (jumpgate, events), structure indicator
396x76px18pxOther player hull icons + usernames
4Dynamic (3x3)20pxDetail view: resources, discovery age. No pan.
Coordinate frame: Row labels (Y) on left, column labels (X) on bottom. Margins: FRAME_LEFT=40px, FRAME_BOTTOM=24px, FRAME_PAD=8px.

Visual features:
• Pulsing player border (sinusoidal alpha, period ~2.5s)
• Bookmark borders (6 colors: green, red-orange, blue, yellow, teal, magenta)
• Staleness rendering: discoveries dim at STALENESS_DIM_HOURS, fade to 0.2 alpha at STALENESS_FADE_DAYS * 24
• Selected sector highlight (3px primary border)
• Hull icons rendered as pixel-art patterns (from HULL_RADAR_PATTERNS)
• Feature dots: cyan (#00BFFF) for jumpgate, red (#FF3333) for distress, magenta (#FF00FF) for other events
• Structure indicator: diamond symbol (◊) in top-left corner, amber colored

Jump animation: 3-phase, 800ms total
1. Glitch phase (200ms): scanline displacement + static noise
2. Slide phase (400ms): grid translates in jump direction
3. Settle phase (200ms): residual glitch fades out

Exported functions:
• drawRadar(ctx, state) — main render function
• calculateVisibleRadius(canvasW, canvasH, zoomLevel) — grid size calculation
• drawGlitchOverlay(ctx, width, height, intensity) — pixel-level scanline displacement

useCanvas (Hook)
packages/client/src/canvas/useCanvas.ts
Signature: useCanvas(draw: (ctx: CanvasRenderingContext2D) => void): RefObject<HTMLCanvasElement>
Shared canvas hook used by both RadarCanvas and QuadMapScreen.
• Handles DPR-aware canvas sizing via ResizeObserver
• Scales context by devicePixelRatio for crisp rendering
• Drives requestAnimationFrame render loop
• Catches render errors silently to prevent render loop death
JumpAnimation
packages/client/src/canvas/JumpAnimation.ts
State machine for sector-jump visual effects.
• JumpAnimationState: active, phase ('glitch'|'slide'|'settle'|'none'), progress (0-1), direction ({dx,dy}), startTime
• Phase durations: glitch=200ms, slide=400ms, settle=200ms (total 800ms)
• createJumpAnimation(dx, dy) — initialize from direction vector
• updateJumpAnimation(state, now) — advance state based on elapsed time
• During slide phase: canvas translates by direction * progress * cellSize
• Glitch overlay intensity scales with phase progress

08. State Management

UI state is managed by Zustand with three slices combined in packages/client/src/state/store.ts:

UISlice (uiSlice.ts)

interface UISlice {
  screen: 'login' | 'game';

09. UX Report

This section documents identified UX inconsistencies, accessibility gaps, and architectural divergences in the current UI implementation. Findings are categorized by severity.

A. Inline Styles vs CSS Classes

SEVERITY: HIGH

Finding: Heavy reliance on inline styles throughout component tree. Nearly all dialog and overlay components (BattleDialog, CombatV2Dialog, BattleResultDialog, StationCombatOverlay, FrequencyMinigame, HelpOverlay, BlueprintDialog) use exclusively inline styles with no CSS classes.

Impact: Inline styles cannot be overridden by color profile changes. Hardcoded color values (#FFB000, #FF3333, #00FF88, #00BFFF) bypass the var(--color-primary) system, meaning dialogs do not respond to color profile changes. For example, BattleDialog hardcodes color: '#FFB000' instead of color: 'var(--color-primary)'.

Affected components: BattleDialog, BattleResultDialog, CombatV2Dialog, StationCombatOverlay, FrequencyMinigame, HelpOverlay, BlueprintDialog, LegendOverlay, DetailViewOverlay, BookmarkBar, BezelKnob, LedDot, SchematicView (in GameScreen).

Recommendation: Extract inline styles to CSS classes in crt.css or a new dialogs.css file. Use CSS custom properties throughout for theme compliance.

SEVERITY: MEDIUM

Finding: Inconsistent font-family declarations. Some components use fontFamily: 'monospace' (BattleDialog, BattleResultDialog) while others use fontFamily: 'var(--font-mono)' (CombatV2Dialog, StationCombatOverlay). The CSS custom property resolves to 'Share Tech Mono', 'Courier New', Courier, monospace.

Impact: Visual inconsistency between V1 and V2 dialogs. BattleDialog falls back to browser default monospace font rather than Share Tech Mono.

B. Touch Accessibility Gaps

SEVERITY: HIGH

Finding: FrequencyMinigame has no touch input method. Frequency adjustment only works via keyboard arrow keys or mouse wheel. The canvas element has an onWheel handler but no touch event handlers (onTouchStart, onTouchMove). On mobile devices, the minigame is effectively unusable.

Recommendation: Add touch drag or on-screen +/- buttons for frequency adjustment. Consider a horizontal slider input as an alternative to the canvas-based interaction.

SEVERITY: MEDIUM

Finding: BezelKnob uses <input type="range"> with writingMode: 'vertical-lr'. While this works on desktop, vertical range inputs have inconsistent behavior across mobile browsers. Some browsers may not render the thumb correctly or may not respond to vertical drag gestures.

Recommendation: Test vertical range inputs on iOS Safari and Android Chrome. Consider a custom knob implementation with touch event handling for reliable mobile behavior.

SEVERITY: MEDIUM

Finding: BookmarkBar right-click action (clear bookmark) via onContextMenu has no touch equivalent. Long-press does not trigger context menu on all mobile browsers consistently.

Recommendation: Add an explicit "clear" button or use a long-press handler with a visual indicator (e.g., fill animation during press).

SEVERITY: LOW

Finding: Mobile tab bar exposes only 8 of 12 monitors. LOG, TRADE, FACTION, and QUESTS are not accessible on mobile. Players on mobile devices cannot view event logs, trade, manage factions, or check quests.

Recommendation: Add a "more" tab or implement a scrollable tab bar to expose all 12 monitors on mobile.
